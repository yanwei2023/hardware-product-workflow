# Roadmap

本文档用于管理人机协同硬件开发流程系统的开发节奏、阶段目标、验收标准和推进记录。每次进入新的开发节点时，应同步更新本文件中的节点状态与推进记录。

## 状态约定

| 状态 | 含义 |
| --- | --- |
| `Planned` | 已规划，尚未开始 |
| `In Progress` | 正在开发或验证 |
| `Blocked` | 因外部条件、设计决策或依赖未满足而暂停 |
| `Done` | 已实现并完成基础验证 |
| `Deferred` | 暂缓，后续重新评估 |

## 当前总体目标

围绕公司内部局域网部署场景，先完成一个可被真实团队试用的硬件产品开发流程管理系统：流程引擎负责阶段门和审计闭环，Agent 负责草稿、检查和证据准备，人类负责审核、批准和最终决策。

## 近期重点

### M5 PostgreSQL 增量事务写入扩展

目标是减少运行期对全量精确镜像的依赖，把更多核心业务写入迁移到原生 PostgreSQL 事务路径，并保持 JSON store、审计事件、通知和业务表之间的一致性。

重点任务：

- 梳理仍未接入增量事务的业务写入路径。
- 为每条写入路径明确事务边界、审计事件、通知副作用和补偿策略。
- 补充增量事务测试，覆盖成功提交、校验漂移、补偿恢复和错误响应。
- 更新运行时状态暴露，让前端和部署人员能明确当前写入后端和可写状态。

验收标准：

- 核心业务写入路径具备独立测试。
- 校验失败时不会留下半提交业务状态、审计记录或通知。
- `npm run release:check` 与 `npm run pilot:check` 通过。
- README 或相关 docs 更新当前运行模式和限制。

### M6 内部试点发布与部署固化

目标是让项目从“开发者可运行”推进到“内部试点可交付”。

重点任务：

- 固化局域网部署路径，包括端口、环境变量、数据目录、备份目录和数据库连接配置。
- 准备试点前检查清单，覆盖服务健康、数据读写、PostgreSQL preflight、工作台访问和回滚包。
- 明确版本发布流程，包括打包、校验、归档、部署和回滚。
- 记录试点期间的问题收集方式和优先级规则。

验收标准：

- 试点包包含可复核校验文件。
- 试点部署文档可由非核心开发人员执行。
- 出现数据或服务问题时，有明确回滚步骤。
- 试点反馈能被记录到统一台账并映射到后续节点。

## 每次迭代更新规则

每次完成开发或验证后，应至少更新以下内容：

- 将对应开发节点的 `当前状态` 调整为最新状态。
- 在 `推进记录` 中追加一行，写明日期、节点、阶段、完成内容和下一步。
- 如果新增了发布、部署、数据迁移或权限相关风险，应同步记录到相关文档或 issue。
- 如果验收标准发生变化，应直接更新对应节点，避免 roadmap 与实际交付脱节。
